## Summary

Build a complete full-stack food ordering application with customer and administrative panels. The backend uses Express and MongoDB (Mongoose ODM) running on port 5000, enforcing stateless JWT authentication and role-based access control. Payments are integrated via the PayHere Sandbox with secure backend MD5 parameter signature calculation and webhook callback verification. The frontend is built with React (Vite) running on port 5173.

## Technical Context

**Language/Version**: JavaScript / ES Modules (Node.js v18+, React 19)

**Primary Dependencies**: React Router Dom v7, Axios, Express v4, Mongoose v9, Bcryptjs v3, Jsonwebtoken v9, Cors, Dotenv

**Storage**: MongoDB (Atlas/Local) queried strictly via Mongoose models

**Testing**: Manual E2E validation of payment redirects, webhook simulations, and role guards

**Target Platform**: Node.js Linux server (Backend) / Web Browser (Frontend)

**Project Type**: Web application (separate client and server modules)

**Performance Goals**: API response latency < 200ms, page load time < 500ms

**Constraints**: Stateless JWT auth only (no session state on server), strict signature validations for PayHere webhook notifications, local dev ports 5000 (server) and 5173 (client)

**Scale/Scope**: Proof-of-concept system containing ~10 client views, ~12 server routes, 4 mongoose models, and 1 core database seed script.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- **Principle I: Stateless JWT Auth & Authorization** -> **PASS**. No server sessions. JWT tokens are verified on-the-fly via custom express middlewares (`auth` and `adminGuard`). Private and Admin routes guard client-side navigation.
- **Principle II: PayHere Sandbox Flow** -> **PASS**. The backend computes checkout hashes securely. The webhook `/api/payments/notify` parses POST parameters, calculates MD5 hashes using merchant secrets, and compares signature hashes prior to data mutation.
- **Principle III: Strict Mongoose Schema and Database Consistency** -> **PASS**. Schemas defined with explicit restrictions. Order item objects snapshotted (name, price, quantity) rather than referencing changing catalog items directly.
- **Principle IV: Clean Separation of Layered Architecture** -> **PASS**. Server structure maps strictly to controllers, routes, models, and middleware. Frontend maps to reusable components, pages, and AuthContext.
- **Principle V: Data Seeding and Development Setup** -> **PASS**. A dedicated `seed.js` script initializes administrative users and food items. Port 5000 serves the API, and port 5173 is configured for client-side Vite proxying.
